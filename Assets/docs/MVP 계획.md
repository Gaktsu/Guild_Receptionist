아래는 네 **“길드 접수원” 기획서**를 기준으로, **빠르게 MVP를 만들고 → 그 위에 확장**하는 전형적인 제작 순서야. (UI 중심, 100일 캠페인, Info→Quest→결과 루프, 정적=SO / 동적=JSON, seed 고정, 디버그 패널 필수 같은 원칙을 그대로 반영함)

---

## MVP 목표를 “한 문장”으로 고정

**Day를 1일 진행하면**

1. 정보 카드가 뜨고(기록/폐기/추가조사) →
    
2. 선택한 정보로 퀘스트를 만들고(템플릿/슬롯) →
    
3. 오늘 제출(최대 N개) →
    
4. 결과(성공/실패 + 수치 로그) →
    
5. 다음날로 넘어간다(자동 저장)
    

이게 “된다”가 MVP 합격선.

---

## MVP에서 **과감히 빼도 되는 것**

빠르게 MVP를 내고 싶으면, 아래는 **처음엔 내부 계산/더미**로 두는 게 좋아.

- **모험가 개별 관리 UI** (고용/해고/성향/장비 등): _MVP에선 파티 등급 분포만_ 있으면 됨
    
- **대형 이벤트의 서사/선택지 장편화**: _이벤트 1~2종만_ 상태머신 검증용으로
    
- **업그레이드/정책 다양화**: _업그레이드 1개(예: 조사실 AP+1)만_
    
- **지도/보조 UI**: MVP에선 없어도 됨

## 시스템(모듈) 제작 순서: “깨지면 안 되는 기반 → 루프 → 확장”

아래 순서로 만들면, 중간에 기능이 늘어도 구조가 덜 흔들려.

### 0) 프로젝트 골격 & 규칙 세팅 (가장 먼저)

- 폴더를 기능 기준으로 고정 (Guild/, Quest/, Event/, Time/, Player/ …)
    
- **정적 데이터 = ScriptableObject**, **동적 진행 = JSON 저장**
    
- “Manager 남발 금지”: 시스템 단위만 Manager 허용
    
- **단일 seed 기반 RNG**(재현성) + 디버그 패널 계획
    

> ✅ 네가 할 일: “폴더/네이밍/금지 규칙/데이터 원칙”을 문서 1장으로 뽑아 Codex에게 항상 붙여넣기(프롬프트 템플릿화)  
> ✅ Codex가 할 일: 빈 프로젝트 + 폴더 + 기본 어셈블리/네임스페이스 + RNG 래퍼 + JSON 저장 뼈대

---

### 1) **Day 진행 스테이트머신 + 화면 흐름** (UI 뼈대)

기획서의 UI Step( Day Start → 정보수집 → 퀘스트작성 → 제출/마감 → 결과/정산 → Next Day )를  
**진짜로 화면 전환**되게 만드는 게 1차 승부처.

- DayState enum + 전환 규칙
    
- 각 스텝 패널(빈 UI) + 버튼 전환
    
- “End Day” 시점에 **저장 호출**
    

> ✅ 네가 할 일: “각 Step에서 플레이어가 누르는 버튼 목록 + 패널 구성”을 체크리스트로 확정  
> ✅ Codex: UI 패널 5개(프리팹) + DayFlowController

---

### 2) **핵심 데이터 모델(도메인) 확정** (MVP의 뼈)

MVP에서 최소로 필요한 데이터만 먼저 만든다.

- **WorldState**: 평판/안정도/예산/세력관계/인명피해 (5개 고정)
    
- **Info(원본)** / **Lead(소모되는 활용권)**
    
- **QuestDraft / QuestIssued / QuestResult**
    
- (선택) Incident: MVP에선 “자동 그룹 id” 정도로 단순화 가능
    

> ✅ 네가 할 일: 각 데이터의 “필수 필드만” 표로 확정 (필드 늘리기 금지)  
> ✅ Codex: C# record/class + JSON 직렬화 + SO 정의

---

### 3) **정보 생성/표시/보관함** (루프의 시작점)

- 하루 정보 6~8개(초반 기준) 생성
    
- 카드 UI: 제목/출처/지역/유형/신뢰도/요약 + 버튼(기록/폐기/추가조사)
    
- 기록한 Info는 **보관함 리스트**로 들어감(필터는 MVP에선 최소 1~2개만)
    

> ✅ 네가 할 일: “정보 카드 10개 샘플”을 직접 작성(톤/길이 규칙 적용) → SO로 넣기  
> ✅ Codex: InfoGenerator(데이터 기반) + InfoCard UI + Archive 저장

---

### 4) **AP(행동 포인트) + ‘추가조사’만 먼저 구현**

AP는 “퀘스트 수 제한”이 아니라 **품질 조정 자원**. MVP에선 행동 1개만 넣어도 충분해.

- AP = 4~6 지급, 남은 건 소멸
    
- “추가조사(비용/AP 1)” → Lead의 credibility 상승 같은 단순 효과
    

> ✅ 네가 할 일: AP 수치(기본 5 추천)와 조사 효과(예: 신뢰도 +15, 최대 100) 확정  
> ✅ Codex: ActionPointSystem + UI 표시 + 조사 버튼 연결

---

### 5) **퀘스트 작성기(템플릿/슬롯)**

MVP는 “완전 자동 생성”보다 **반자동(템플릿 + 슬롯 채우기)**가 빠르고 안정적이야.

- 템플릿 6개만: 토벌/조사/호위/배송/정화/외교
    
- Info(또는 Lead)를 슬롯에 붙이면
    
    - 타입 추천 3개
        
    - 위험/보상/마감이 계산되어 미리보기 표시
        
- “제출 목록에 추가” 버튼
    

> ✅ 네가 할 일: 템플릿 6개의 “슬롯 정의”를 먼저 고정 (대상/장소/위험/마감…)  
> ✅ Codex: QuestEditor UI + Template SO + QuestCalculator(룰/테이블)

---

### 6) **오늘 제출(최대 N개) + 결과 시뮬레이션(성공/실패)**

- 오늘 제출 리스트(최대 4개 추천)
    
- 결과는:
    
    - 큰 결과(성공/실패/부분성공)
        
    - 수치 변화(평판/예산/안정도 등)
        
    - 이유 Top3 로그(“신뢰도 낮음”, “마감 촉박” 등)
        

시뮬레이션은 MVP에선 단순하게:

- 파티 등급(랜덤 분포) vs 퀘스트 등급(계산 결과)
    
- credibility/urgency 보정
    
- seed 기반 RNG로 결과 재현
    

> ✅ 네가 할 일: “성공 확률 테이블(등급차)”과 보정폭(신뢰도/긴급도)을 문서에서 그대로 확정  
> ✅ Codex: ResolveQuest() + Result UI + Log 생성

---

### 7) **이벤트 상태머신 1종만 연결**

MVP에서 이벤트는 “콘텐츠”가 아니라 **시스템 검증 장치**야.

- Dormant → Triggered → Escalating → Critical → (Resolved/Catastrophe)
    
- 트리거: 최근 5일 실패 2회 같은 단순 조건 1개만
    
- 이벤트가 월드 상태에 영향을 줘서 다음날 정보에 반영(가짜라도)
    

> ✅ 네가 할 일: 이벤트 1개(예: 역병)만 “조건/영향/페이즈 텍스트”로 정의  
> ✅ Codex: EventSystem + Phase 전이 + 디버그 강제 전이 버튼

---

### 8) **엔딩(100일) / 즉시 게임오버만 최소 구현**

- 즉시 종료 1~2개(예: 평판 0, 예산 -1000 이하)
    
- 100일 도달 시 기본 엔딩 1개(추후 확장)
    

> ✅ 네가 할 일: “MVP 엔딩 조건 3개”만 고정  
> ✅ Codex: EndingEvaluator + 결과 화면

## 너(설계자)가 “순서대로” 해야 하는 일 체크리스트

### A. 하루~이틀 안에 해야 하는 설계 확정(가장 중요)

1. **MVP 스코프 선언문**(위의 한 문장 루프 + 뺄 것 목록)
    
2. **데이터 모델 표(필수 필드만)**: WorldState / Info / Lead / Quest / Result
    
3. **템플릿 6개 슬롯 정의**
    
4. **수치 테이블 3개만 확정**
    
    - 등급차 성공확률
        
    - 신뢰도 보정
        
    - 긴급도(마감/리스크) 보정
        
5. **이벤트 1개 정의서**(트리거/영향/페이즈 텍스트)
    

### B. Codex에게 맡길 때의 운영 규칙(속도 폭발함)

- “한 번에 1~3 파일만 생성/수정”
    
- “핵심 로직은 순수 함수 + 상태 객체”
    
- “seed는 반드시 외부에서 주입”
    
- “테스트 1개는 최소 포함(확률/이벤트 전이 같은 핵심만)”
    
- “전역 상태/매니저 난발 금지”
